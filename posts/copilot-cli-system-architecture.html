Copilot CLI System Architecture Deep Dive

GitHub Copilot CLI is not a thin wrapper around a model API — it is a layered runtime with distinct responsibilities at each level. Understanding how the six architectural layers collaborate gives you an accurate mental model for debugging unexpected behaviour, tuning performance, and extending the CLI with plugins, MCP servers, and custom agents.

Availability: All GitHub Copilot paid plans (Pro, Pro+, Business, and Enterprise).

The six architectural layers

Each layer has a clearly bounded responsibility and communicates with adjacent layers through defined interfaces. The table below shows the six layers from top (user-facing) to bottom (infrastructure):

Layer Primary responsibility Key components
User Interface Render input/output and handle user interaction Input handler, timeline renderer, status bar, permission dialogs
Session Management Persist conversation state and compact context automatically Session store (~/.copilot/session-state/), compaction loop, checkpoints
Agent Execution Orchestrate agent lifecycle and delegation patterns Main agent, subagents via read_agent, background delegation (& / /delegate)
Tool & Extension Provide tools, MCP integrations, plugins, skills, and LSP servers Built-in tools, MCP servers, plugin system, skills, LSP integration
Model & API Route requests to the right model and stream responses Model router, Copilot API client, streaming, retry/error handling
Permission & Security Authenticate the user and gate file/tool access Token resolver, permission manager, parallel approval, path allowlist

User Interface layer

The UI layer adapts its rendering to three distinct modes. Choosing the right mode can meaningfully improve your experience in constrained terminals or for accessibility needs:

Mode Flag Best for
Standard Terminal (default) CI pipelines, SSH sessions, any terminal
Alt-Screen --alt-screen Full-screen diffs (/diff), mouse selection, scrollable history
Screen Reader --screen-reader Accessibility-optimised output — static text replaces spinners

The timeline renderer displays the full conversation history including expandable tool executions. The status bar surfaces the current working directory, git branch, selected model, and IDE connection status — all without interrupting the active prompt.

Session Management layer

Sessions are persisted to disk in a structured format that cleanly separates storage from presentation. The new format lives under ~/.copilot/session-state/; a legacy format under history-session-state/ is migrated automatically on resume. 

~/.copilot/
├── session-state/            # Current session format
│   └── <session-id>/
│       ├── metadata.json     # Model usage, token counts, PRU, file changes
│       ├── messages/         # Full conversation history
│       └── checkpoints/      # Rewind points (Esc Esc to access)
└── history-session-state/    # Legacy format (auto-migrated on --resume)

The compaction loop is the mechanism that enables infinite sessions. When the context window reaches 95 % of the model's token limit, the runtime automatically compresses the conversation history in the background and re-injects a summary — without pausing your work. Any messages sent during compaction are queued and delivered once compaction finishes.

Compaction behaviour Detail
Trigger threshold 95 % of the active model's token limit
What is preserved Tool call sequences, file edit history, reasoning summaries (for models that support extended thinking)
Rewind checkpoints Snapshots saved before each compaction so you can restore the full uncompressed history — press Esc Esc to access them interactively
Manual trigger /compact command
Message queuing Messages sent mid-compaction are automatically queued and replayed

Agent Execution layer

Agents are defined in Markdown files with YAML frontmatter. The frontmatter declares which tools the agent may use, whether it can invoke the model, and any agent-specific MCP server configurations: 

---
name: my-agent
description: A focused code-review agent
tools: [view, grep, glob, shell]
infer: true
---
You are a careful code reviewer …

The layer supports three delegation patterns that control whether execution is synchronous, delegated to a subagent, or fully backgrounded:

Pattern How to invoke Blocks the main agent?
Switch agent /agent <name> or --agent <name> flag Yes — replaces the active agent for the session
Subagent delegation read_agent tool (called by the model) Yes — main agent waits for the subagent result
Background execution /delegate <task> or prefix the message with & No — runs concurrently; results appear in the timeline

Tool & Extension layer

This layer is the richest in the architecture. It exposes built-in tools, Model Context Protocol (MCP) server integrations, a plugin system, reusable skills, and LSP-based code intelligence — all composable with each other.

Built-in tools cover the most common agent actions:

Tool Purpose Permission required
view / view_range Read file contents (full or range) Read path approval
edit Modify file contents Write path approval
shell Execute shell commands Command approval + path
grep / glob Search codebase (uses bundled ripgrep) Read path approval
web_fetch Fetch web content URL permission
apply_patch Apply unified diff patches Write path approval
store_memory Persist facts to repository memory Memory enabled
read_agent Delegate to a subagent None
ask_user Request clarification interactively None

Plugins extend the CLI with custom agents, bundled MCP servers, skills, and LSP configurations. They live under ~/.copilot/plugins/<plugin-name>/ and are managed with /plugin install, /plugin update, and /plugin uninstall.

Skills are reusable Markdown prompt templates invocable as slash commands (e.g. /my-skill). They survive compaction — the skill definition is re-injected into the compacted context — making them reliable across long-running sessions.

Model & API layer

The model router decouples the agent from any specific AI provider. You switch models with /model <name> and the selection persists across sessions. The CLI supports both Claude and GPT model families, with per-model tracking of token consumption and Premium Request Usage (PRU) multipliers.

The API client communicates with per-subscription endpoints (enabling data-residency requirements), streams responses token-by-token, and retries automatically on server errors. You can disable streaming with --stream off or route requests through a proxy via the HTTPS_PROXY environment variable.

Permission & Security layer

Authentication uses a five-step token resolution chain, evaluated in order:

  1. COPILOT_GITHUB_TOKEN environment variable
  2. GH_TOKEN environment variable
  3. GITHUB_TOKEN environment variable
  4. gh CLI authentication store
  5. GITHUB_ASKPASS script

The permission manager gates every file system operation and external tool call. By default the allowlist covers the current working directory and the system temp directory; every other path requires explicit user approval.

Flag Effect
--allow-all / --yolo Auto-approve all permissions for the session
--allow-all-paths Auto-approve all file-path access (prompts still appear for other tool types)
--disable-parallel-tools-execution Require sequential approval — no concurrent tool calls

When multiple tools execute in parallel (the default), the first approval of a given permission type with "Approve for session" automatically resolves all pending requests of the same type — dramatically reducing approval fatigue in large agentic tasks.

How the layers interact: a control-flow walkthrough

The following pseudocode traces a single user message from input to response, showing which layer handles each step: 

1. UI layer        — receives multi-line input; renders timeline entry
2. Session layer   — appends message to disk; checks context window usage
3. Agent layer     — selects active agent; assembles system prompt + instructions
4. Permission layer — verifies authentication token is valid
5. Model layer     — sends context window to Copilot API; streams response tokens
6. Agent layer     — parses model output; detects tool-call requests
7. Permission layer — prompts user to approve each tool (unless pre-approved)
8. Tool layer      — executes approved tools concurrently; collects results
9. Session layer   — appends tool results to context; checks 95 % threshold
                     └─ if threshold reached → compaction loop fires →
                        compress history → re-inject summary → continue
10. Model layer    — re-invokes model with updated context (ReAct loop)
11. UI layer       — renders final response and updated timeline
Key insight: Steps 6–10 form the ReAct loop (Reason → Act → Observe → Reason…). The model is not called once per user message — it is called once per reasoning step, which may involve dozens of tool invocations before producing a final response.

Configuration hierarchy

The CLI resolves configuration from multiple files with a defined precedence. Lower entries in the table are overridden by higher entries at runtime:

File Scope Purpose
~/.copilot/config User-global Persistent CLI settings (log level, banner, screen reader, experimental)
~/.copilot/mcp-config.json User-global Global MCP server definitions
~/.copilot/lsp-config.json User-global User-level LSP server configurations
.vscode/mcp.json Workspace Workspace-scoped MCP servers (shared via source control)
.github/lsp.json Repository Repository-level LSP server configurations
.github/copilot-instructions.md Repository Custom instructions injected into every session in this repo
.copilot-instructions.md Repository root Alternative location for custom instructions
