Copilot CLI System Architecture Deep Dive

GitHub Copilot CLI is more than a thin wrapper around an LLM. It is a layered agent runtime whose components — the entry-point, session engine, tool subsystem, context manager, customization layer, and delegation bridge — all collaborate in well-defined ways. Understanding that architecture helps you predict behavior, diagnose failures, and extend the CLI purposefully.

Availability: All GitHub Copilot paid plans (Pro, Pro+, Business, and Enterprise).

High-level component map

At the outermost level the CLI exposes two interfaces that converge on the same internal agent loop: 

┌─────────────────────────────────────────────┐
│               GitHub Copilot CLI            │
│                                             │
│  ┌──────────────┐    ┌─────────────────┐    │
│  │  Interactive  │    │  Programmatic   │    │
│  │  interface   │    │  interface      │    │
│  │  (copilot)   │    │  (copilot -p …) │    │
│  └──────┬───────┘    └────────┬────────┘    │
│         └──────────┬──────────┘             │
│                    ▼                        │
│          ┌──────────────────┐               │
│          │   Agent Loop     │               │
│          │  (ReAct engine)  │               │
│          └────────┬─────────┘               │
│      ┌────────────┼─────────────┐           │
│      ▼            ▼             ▼           │
│  Session      Tool           Context        │
│  Engine      Subsystem       Manager        │
│      │            │             │           │
│      ▼            ▼             ▼           │
│  ~/.copilot/  shell/write/   compaction     │
│  session-     MCP/Skills     + token        │
│  state/                      budget         │
└─────────────────────────────────────────────┘

Entry point: two interfaces, one loop

The CLI boots in one of two modes, both of which feed the same agent loop:

Interface How to invoke What it does
Interactive copilot Opens a REPL-style session. You type prompts, Copilot responds and may call tools. Supports slash commands, plan mode (Shift+Tab), inline tool-approval UI, and context visualization.
Programmatic copilot -p "…" or piped input Runs a single prompt non-interactively and exits. Designed for scripts, CI pipelines, and automation. Use --allow-tool or --allow-all-tools to skip approval prompts.

The agent loop (ReAct engine)

The core of the CLI is a ReAct loop — the model Reasons, takes an Action (tool call), Observes the result, and Reasons again. This cycle continues until the model emits a final response instead of another tool call. 

┌──────────────────────────────────────────────┐
│                  Agent Loop                  │
│                                              │
│  User prompt ──► Context Window ──► LLM      │
│                       ▲              │       │
│                       │         ┌────▼────┐  │
│                       │         │ Tool    │  │
│                       │         │ call?   │  │
│                       │         └────┬────┘  │
│                       │    Yes       │       │
│                       │◄─── result ──┘       │
│                       │                      │
│                    No │ ──► Final response    │
│                       │     to user          │
└──────────────────────────────────────────────┘

The loop runs entirely inside a single OS process. Tool calls are synchronous from the model's perspective — execution pauses, the tool runs, its output is appended to the context window, and the model resumes generation.

Session engine

The session engine owns durable state. Every conversation is backed by a directory on disk: 

~/.copilot/session-state/{session-id}/
├── events.jsonl      # Append-only log of every turn
├── workspace.yaml    # Session metadata (model, dates, repo path)
├── plan.md           # Structured plan, if one was created
├── checkpoints/      # Compaction snapshots (one per compaction cycle)
└── files/            # Temporary artifacts produced by the session

Key responsibilities of the session engine:

  • Create a fresh session directory when a new conversation starts.
  • Persist every event (user message, tool call, tool result, model response) to events.jsonl as it happens — enabling crash recovery and forensic inspection.
  • Resume an existing session (via /resume or --session-id) by replaying its events into the context window.
  • Hand off to the cloud coding agent when /delegate is used — the session ID travels with the delegation so the agent can write back results.

Tool subsystem

The tool subsystem is the agent's effector layer — it translates LLM intent into real-world actions. Tools are organized into four families:

Family Examples Approval model
Shell bash, git, npm run test Per-command approval (or allowlisted via --allow-tool)
File I/O write, read, patch Scoped to trusted directories; write requires approval
MCP servers GitHub MCP, custom MCP servers, third-party integrations Configured in ~/.copilot/mcp.json; each tool exposed by the server follows the same approval flow
Skills Built-in CLI skills, custom skills defined in .github/skills/ Run as sub-agents; the parent agent delegates and awaits the result

When the LLM emits a tool-call JSON object, the tool subsystem:

  1. Checks whether the tool is on the session allowlist.
  2. If not, surfaces an approval prompt to the user (interactive mode) or blocks (programmatic mode without --allow-tool).
  3. Executes the tool in a subprocess scoped to trusted directories.
  4. Appends the result — including stdout, stderr, and exit code — back into the context window as a tool result message.
  5. Returns control to the agent loop, which feeds the augmented context to the LLM.

Context manager

The context manager sits between the session engine and the LLM call. Its job is to fill the context window optimally and ensure it never overflows. 

Context Window (e.g. 200 k tokens)
┌──────────────────────────────────────────┐
│  System prompt + custom instructions     │  ~5 %
│  Tool definitions (shell, MCP, skills)   │  ~5 %
│  Conversation history (user + model)     │  variable
│  Tool results                            │  variable
│  ─────────────────────────────────────── │
│  Free space + compaction buffer          │  ≥5 %
└──────────────────────────────────────────┘

When the conversation approaches 95% of the token limit, the context manager triggers auto-compaction:

  1. It snapshots the current context to a new checkpoint directory under checkpoints/.
  2. It sends the full history to a summarization call — a separate LLM invocation whose sole job is to produce a dense summary.
  3. The summary replaces the raw history in the context window; verbatim turns beyond the summary are discarded.
  4. Execution resumes seamlessly — the model sees only the summary + recent turns, with no interruption to the user.

This is what enables infinite sessions: the context window never truly fills because old content is continuously compressed and replaced.

Developer tip: Use /context to inspect the live token budget breakdown — system tokens, history tokens, tool definitions, and free space. Use /session checkpoints to review what was compressed and when.

Customization layer

All customization is resolved at session startup and injected into the system prompt before the first LLM call. The CLI merges instructions from multiple locations in a fixed discovery order:

Location Scope Priority
~/.copilot/copilot-instructions.md Global (all sessions) Lowest
.github/copilot-instructions.md Repository Medium
.github/instructions/**/*.instructions.md Repository (modular) Medium
AGENTS.md / Copilot.md Repository (root or cwd) High

All matching files are merged (not overwritten) into a single combined instruction set. Repository instructions take precedence over global ones when there are conflicts.

Beyond static instructions, the customization layer also registers:

  • MCP servers — declared in ~/.copilot/mcp.json; their tools appear in the tool subsystem alongside built-in tools.
  • Custom agents — specialized sub-agents the orchestrating model can delegate to (e.g., a frontend-focused agent that enforces team style).
  • Hooks — shell commands that fire at defined lifecycle events (pre-tool, post-tool, pre-response) for logging, security scanning, or CI integration.
  • Copilot Memory — a persistent key-value store for repository conventions that the runtime injects into the system prompt automatically, reducing repetitive context-setting across sessions.

Delegation bridge

The /delegate command hands work off to the cloud coding agent, allowing you to continue working locally while an asynchronous task runs on GitHub's infrastructure. The flow looks like this: 

Local CLI                      GitHub Cloud
─────────────────────          ──────────────────────────
User types /delegate ──────►  Coding agent spins up
                               ├── reads repo context
                               ├── runs ReAct loop
                               ├── commits changes
                               └── opens pull request
                    ◄──────── Session ID + PR link returned
User continues locally
(can track with gh agent-task view)

The session ID created locally is echoed to the cloud agent, so both sides share a common identity for tracking. Once the PR is open, you can resume or steer the session from any of the 7 tracking surfaces (GitHub UI, GitHub CLI, VS Code, JetBrains, Eclipse, Raycast, GitHub Mobile).

End-to-end data flow: a worked example

Tracing the message "Add input validation to the login form" through every component: 

1. Entry point receives the prompt (interactive interface).

2. Session engine writes:
   { "type": "user", "content": "Add input validation…" }
   → events.jsonl

3. Context manager assembles the window:
   [system prompt + merged instructions]
   [tool definitions: shell, write, MCP …]
   [conversation history from events.jsonl]
   [new user message]

4. Agent loop sends assembled context to the LLM.

5. LLM responds with a tool call:
   { "tool": "shell", "input": { "cmd": "grep -r 'login' src/" } }

6. Tool subsystem:
   a. Checks allowlist → not present → surfaces approval prompt.
   b. User approves "Yes, and approve shell(grep) for this session".
   c. Runs grep in a subprocess; captures stdout.
   d. Appends tool result to context window.

7. Agent loop re-enters: LLM now sees grep output.

8. LLM calls write tool to patch src/components/LoginForm.tsx.

9. Tool subsystem executes write; appends result.

10. LLM calls shell tool: "npm run test".

11. Tests pass → LLM emits final response to user.

12. Session engine appends model response to events.jsonl.
    Context manager checks token usage → 42 % used → no compaction.
